# Stage 3 (Production trust) — prioritized plan

**Date:** 2026-07-17 (cron run 4)
**Status:** proposed — awaiting Luca's sign-off (Needs Luca item 10 in
`km/notes/roadmap-cron-state.md`). Per the stage prompt's default ("auth first if
nothing from launch says otherwise" — launch hasn't happened, so nothing does),
execution starts on item 1 while the sign-off is pending; reordering later costs
nothing because the items are independent until the liveness pair.

## Repo facts the plan rests on (verified this run)

- No liveness machinery exists: no heartbeats, no `mesh.health.*` / `mesh.death.*`
  publishing anywhere in `src/` (grep clean). ADR-0016's "heartbeats become a
  secondary mechanism" assumes a primary that was never built.
- No auth surface: `AgentMesh.__init__` takes only `url`; no `creds`/TLS params,
  no `connection_denied` in `_errors.py`.
- ADR statuses: 0038 spec (has code sample — buildable), 0016 spec (no code
  sample — needs amendment before tests, per the pipeline rule), 0040 discussion,
  0048 discussion, 0055 spec (has code sample — buildable, independent).
- No open GitHub issues; launch is still unpublished, so no launch feedback to
  fold in.

## Order

1. **ADR-0038 auth** — gates every serious deployment; also unblocks 0016/0048's
   `$SYS.>` needs (system-account access is an auth-config concern). Sliced:
   - **A. SDK surface:** `creds=`, `tls_cert/tls_key/tls_ca`, `OAM_CREDS` env,
     `.oam-url` TOML `creds` field, `connection_denied` error code. Tested
     against a real JWT-auth'd nats-server using checked-in test credentials
     (generated once with nsc; static, test-only).
   - **B. `oam auth` CLI:** `init`, `user add` (role templates worker/invoker/
     observer), `user revoke`, `whoami` — wrapping nsc per the ADR.
   - **C. Docs:** auth concepts page + the "secured multi-node mesh" cookbook
     recipe (a stage exit criterion), with executable twin.
2. **ADR-0016 + ADR-0040 liveness pair** — shared machinery, done together.
   First amend 0016 (add code sample; decide where the health monitor lives —
   proposal: the `oam mesh up` process hosts it, SDK publishes heartbeats) and
   shape 0040 discussion→spec (pessimistic single-instance fast-fail; skip
   multi-instance until correlation exists, as the ADR itself recommends).
   Ends with the chaos-style test (kill agent mid-request, caller gets
   `agent_died` sub-second instead of a timeout) — a stage exit criterion.
3. **ADR-0048 observability** — shape discussion→spec first; propose a trimmed
   v1: logs subjects + KV-based level control + `oam observe logs`. Traces and
   `$SYS.>` bridging only if v1 lands fast (bridging depends on 0038's system
   account posture).
4. **ADR-0055 lifecycle gates** — spec-ready and independent; last because it
   hardens DX rather than trust. If a run finds itself blocked on liveness
   design questions, this is the parallel workstream to pick up.
5. **ADR-0056 admin UI** — only if Luca confirms the Stage-2 deferral (Needs
   Luca 9); slots after the liveness pair so the registry screen's liveness
   indicator uses real disconnect advisories (see roadmap-learnings 2026-07-17).

**Deferred, with reasons:** ADR-0006 SLA gating (didn't fall out of the MCP
bridge trivially in Stage 1; nothing in Stage 3 needs it); multi-account
tenancy, auth callout, sidecar data plane (all explicitly out of scope in
ADR-0038); OTel export (0048's ephemeral-subjects layer comes first by design).

  which Luca can review as a normal ADR diff.
- 0038 open question "TLS posture in `oam auth init`": proposal — generate a
