# ADR-0038: NATS authentication and credential management

- **Type:** architecture
- **Date:** 2026-04-18
- **Status:** spec
- **Related:** ADR-0033 (CLI surface, auth deferred), ADR-0035 (control plane), ADR-0037 (OAM scope), ADR-0022 (`AgentMesh.local()`)
- **Source:** conversation (shaping session on authn/z)

## Context

ADR-0033 explicitly deferred authentication for Phase 1: `oam mesh connect` targets open meshes, no credentials, no transport security. This is correct for local development but unshippable for any mesh reachable outside a trusted network. The DX choice deferred then was "pick the auth story later, once we understand what we need".

The authn/z shaping session determined that NATS offers enough primitives (NKey + JWT, accounts, per-subject permissions, mTLS) that OAM should not invent its own. The remaining design work is about **DX over NATS auth**: how does an agent author or operator enable auth, where do credentials come from, what does the SDK surface look like, and how does this compose with ADR-0037's OAM scope.

This ADR covers the trust boundary (who is this NATS client, what may their connection touch). ADR-0037 covers logical scoping inside that boundary (which agent can see or call which). They compose: scope works without NATS auth on, NATS auth works without scope, both together is the production configuration.

## Decision

### 1. Position on the NATS auth menu

OAM adopts **NKey + JWT (decentralized auth)** as the production default. Token and username/password are documented as valid "put a lock on the door" options for internal meshes but are not the recommended path. mTLS is an optional transport-layer addition, orthogonal to JWT identity.

Reasoning: NKey + JWT is the path NATS itself recommends for non-trivial deployments. Credentials ship as a single `.creds` file, servers verify cryptographically without a database, revocation and rotation are first-class, and the tooling (`nsc`) already exists. Anything we might invent would be a worse version of this.

### 2. Local dev stays open

`AgentMesh.local()` (ADR-0022) and `oam mesh up` continue to start an open mesh with no credentials. Hello-World remains unchanged (ADR-0008 DX-first). Auth is only a concern when pointing at a remote mesh.

### 3. Identity granularity: per-process + per-human

OAM distinguishes two kinds of identity, both generated by the same tooling:

| Kind | Who holds it | Permissions scope | Example |
|------|--------------|-------------------|---------|
| **Workload** | a running OAM process (may host several agents) | union of subjects needed by the agents it hosts | `risk-pipeline-prod` |
| **Operator** | a human operator running the `oam` CLI | read-only access to catalog, health, control-plane surfaces | `luca-dev` |

A single NATS connection is one identity. Per-agent identity is explicitly rejected: it would force one connection per agent, explode credentials and deployment complexity, and delivers no capability that the per-agent OAM scope layer (ADR-0037) does not already provide logically.

### 4. Credential DX: `oam auth init`

OAM does not build its own issuer service. It wraps `nsc`, NATS's security-credentials CLI, and emits credential trees aligned with OAM's subject layout.

```bash
# Interactive bootstrap: creates an operator, an account, and initial users.
# Writes credentials into .oam/creds/ and the operator JWT push URL into .oam-url.
oam auth init

# Non-interactive generation of a workload credential with a role template.
oam auth user add risk-pipeline --role worker --channel finance.risk
# -> writes ./risk-pipeline.creds

# Operator (human) credential.
oam auth user add luca --role observer
# -> writes ./luca.creds

# Revoke and push updated account JWT to the resolver.
oam auth user revoke risk-pipeline
```

Output is a standard NATS `.creds` file usable by any NATS client, not just OAM. We add value by making the role templates fit OAM's subject conventions, not by locking users in.

### 5. Role templates

`oam auth init` ships three coarse role templates. They map cleanly onto OAM's subject hierarchy (`docs/architecture/subjects.md`).

| Role | Intent | NATS publish allow | NATS subscribe allow |
|------|--------|--------------------|----------------------|
| `worker` | hosts agents, emits events, serves invocations | `mesh.health.>`, `mesh.agent.*.*`, `mesh.agent.*.*.events`, `mesh.errors.>`, `_INBOX.>`, `$JS.API.>`, `$KV.mesh-registry.>` (own-key only, enforced by policy template) | `mesh.agent.>`, `_INBOX.>`, `$KV.mesh-catalog.>`, `$KV.mesh-registry.>` |
| `invoker` | calls agents, reads the catalog | `mesh.agent.>`, `_INBOX.>` | `_INBOX.>`, `mesh.results.>`, `$KV.mesh-catalog.>`, `$KV.mesh-registry.>` |
| `observer` | read-only: catalog, health, events | (none beyond `_INBOX.>`) | `$KV.mesh-catalog.>`, `$KV.mesh-registry.>`, `mesh.health.>`, `mesh.agent.>.events` |

Roles are coarse-by-design. A process that hosts multiple agents takes `worker`, gaining publish rights on `mesh.agent.*.*`. The tighter "this process may only serve `finance.risk.scorer`" property belongs to the OAM scope layer (ADR-0037), not to NATS-level permissions, because enforcing it at NATS forces one process per agent.

Users may extend a role with per-user allow/deny overrides emitted into the user JWT. Templated roles with `{channel}` / `{name}` placeholders are a possible future extension; coarse-only is the Phase 2 default.

### 6. Tenancy: single account for now

One NATS account. Multi-account tenancy is deferred. The accounts primitive remains available for operators who want it, but OAM does not assume it. Cross-tenant stories (shared observability, cross-tenant catalog, federation) are out of scope until there is a concrete use case.

### 7. SDK surface

`AgentMesh` accepts credentials via a single optional parameter:

```python
AgentMesh(url="nats://mesh.company.com:4222", creds="./agent.creds")
```

Resolution order:

1. Explicit `creds=` argument.
2. `OAM_CREDS` environment variable.
3. `creds` field in `.oam-url` (extended from ADR-0033's "just a URL" format into a small TOML).
4. No credentials: connect open. SDK raises immediately if the server rejects anonymous connections.

mTLS parameters (`tls_cert=`, `tls_key=`, `tls_ca=`) are accepted alongside `creds=` and passed through to `nats-py`. They are orthogonal to auth identity: `creds=` alone yields JWT auth over plaintext (dev) or TLS-encrypted (prod) transport; adding `tls_cert=` / `tls_key=` enables mTLS for transport identity on top.

### 8. Scope composition (ADR-0037)

ADR-0037's OAM scope is caller-side and unaffected by this ADR's credentials. When both layers are active:

- NATS auth gates whether the process's connection may touch a subject at all.
- OAM scope gates which agents the SDK will route to or surface.
- A denial at either layer surfaces as a typed error. NATS rejections bubble up as `MeshError` code `connection_denied` (from the server's permission violation); scope rejections surface as `scope_denied` (ADR-0037). Same exception hierarchy, distinguishable codes.

When NATS auth is on, the server knows the authenticated user for each connection. A future minor extension may surface this via `mesh.connection_identity` for observability; it is **not** consumed by scope enforcement (scope stays caller-side and cooperative).

### 9. CLI impact on ADR-0033

`oam mesh connect` gains optional credential arguments:

```bash
oam mesh connect nats://mesh.company.com:4222 --creds ./luca.creds
```

Persisted in `.oam-url`. Subsequent `oam` commands in that directory use the stored credentials. `oam auth whoami` reports the current identity.

## Code sample (DX contract)

```python
# Workload: an agent process authenticated with a worker credential.
from openagentmesh import AgentMesh, AgentSpec, AgentScope

mesh = AgentMesh(
    url="nats://mesh.company.com:4222",
    creds="./risk-pipeline.creds",   # generated by `oam auth user add`
    scope_default="deny",
)

spec = AgentSpec(
    name="scorer",
    channel="finance.risk",
    description="Scores risk for incoming trades.",
    scope=AgentScope(
        can_receive_from=["orchestrator.*"],
    ),
)

@mesh.agent(spec)
async def scorer(req: RiskRequest) -> RiskScore:
    ...
```

```bash
# Operator: a human using the CLI from their laptop.
oam auth user add luca --role observer
oam mesh connect nats://mesh.company.com:4222 --creds ./luca.creds
oam mesh catalog                     # works: observer allowed to read catalog
oam mesh invoke finance.risk.scorer  # fails: observer may not publish invocations
```

```python
# Local dev: no credentials, no ceremony. ADR-0022 behavior preserved.
async with AgentMesh.local() as mesh:
    @mesh.agent(AgentSpec(name="hello", description="..."))
    async def hello(req: Greeting) -> Reply:
        return Reply(text=f"hello, {req.name}")
```

## Deployment notes

A minimal server-side config for a JWT-auth'd OAM mesh:

```conf
operator: /etc/oam/operator.jwt
resolver: {
  type: full
  dir: /var/lib/oam/jwt_store
  allow_delete: false
  interval: "2m"
}

tls {
  cert_file: "/etc/oam/server-cert.pem"
  key_file:  "/etc/oam/server-key.pem"
}
```

`oam auth init --server-config` can emit this file. System account setup for `$SYS.>` monitoring is left to operator choice; OAM does not prescribe it in Phase 2.

## Non-goals

- **OAM-native identity abstraction.** No `AgentIdentity` type, no issuer service, no OAM-format tokens. We ship NATS-standard `.creds` and let the ecosystem do the rest.
- **Per-agent credentials.** Process-level only. Per-agent logical isolation is covered by ADR-0037.
- **Multi-account tenancy.** Single account for Phase 2. Tenancy via NATS accounts is a future ADR when a concrete use case appears.
- **BYOK for downstream API keys.** ADR-0023 already excluded this. Confirmed: this ADR covers mesh access only.
- **Kill-switch runtime revocation.** Deferred to the ADR-0035 control plane. First iteration of auth is "re-issue credentials, restart affected processes".
- **Templated role generation with `{channel}` / `{name}`.** Coarse roles only in Phase 2; templates are a possible future extension.
- **Credential rotation automation.** Operator rotates manually via `oam auth user add` + process restart. Automated rotation is a later concern.

## Consequences

- A new top-level `oam auth` CLI namespace with `init`, `user add`, `user revoke`, `whoami`.
- `AgentMesh` constructor gains `creds`, `tls_cert`, `tls_key`, `tls_ca` parameters.
- `.oam-url` evolves from a plain string file into a small TOML with `url` and optional `creds` fields. Backwards-compatible: a bare URL is still valid input.
- `MeshError` gains a `connection_denied` code for NATS permission violations. Composes with ADR-0037's `scope_denied`.
- Documentation adds an "Auth" section covering the three identity tiers (open / password / JWT + optional mTLS), role templates, and the composition with scope.
- Phase 2 scope: SDK passthrough + `oam auth init` wrapper. Phase 3+ may add templated roles, multi-account tenancy, runtime revocation integration with ADR-0035.

## Alternatives Considered

**OAM-native identity with our own issuer.** Rejected for Phase 2. Significant machinery (key issuer, token format, verification library) for a problem NATS already solves with battle-tested primitives. Revisit only if a use case demands identity semantics NATS cannot express.

**Per-agent NATS users.** Rejected. Forces one connection per agent, ballooning credential management and deployment cost for a property (per-agent policy) that the scope layer delivers logically without credential explosion.

**Username/password as the recommended default.** Rejected. Static passwords do not rotate well, cannot revoke individual clients without coordinating resets, and provide no audit trail beyond "user X connected". NKey + JWT is only marginally more complex to set up via `oam auth init` and gains all of the above. Passwords remain documented as an acceptable lighter-weight option for single-team internal meshes.

**Accounts-per-tenant at Phase 2.** Rejected. Premature: no concrete multi-tenant deployment requirement today. Accounts remain available to operators who configure them manually; OAM documents the pattern but does not build for it yet.

**Federated resolver (delegated JWT verification).** Out of scope. `resolver: full` with the JWT store on disk is sufficient for single-site meshes. Federated / leaf-node topologies will need their own ADR.

## Open Questions

- **TLS posture in `oam auth init`.** Should the CLI generate a self-signed server cert and CA by default, or require the operator to provide their own? Self-signed is convenient for preview deployments but risks being shipped to production by accident.
- **`.creds` file discovery.** `OAM_CREDS` env var + `.oam-url` TOML field covers most cases. Should there be a conventional `~/.oam/creds/default.creds` fallback for the human-operator case? Probably yes; confirm during implementation.
- **System account exposure.** Observability over `$SYS.>` is powerful for debugging a mesh. Does `oam auth init` create a system account and an associated `sysadmin` role template, or leave that to the operator? Leaning "yes, and document it".
- **Interaction with ADR-0035 control plane.** Runtime revocation at the control-plane level requires account JWT re-push. First iteration of the control plane may sidestep this by operating entirely at the OAM scope layer. To be resolved when ADR-0035 moves from `discussion` to `spec`.
